## 1. Context and Schema Rules Summary

### Agent Schema (from meta-agent-v3.md)

**Agent Types:**
1. **Simple Agent** - Standalone tool, no coordination, direct execution
2. **Worker** - Executes tasks from plan files (5 phases: Read Plan → Execute → Validate → Report → Return)
3. **Orchestrator** - Coordinates multi-phase workflows via Return Control pattern (NO Task tool invocation)

**Required YAML Frontmatter Fields:**
```yaml
---
name: {agent-name}           # kebab-case, unique
description: {clear purpose} # Use proactively for...
color: {color}              # blue|cyan|green|purple|orange
# Optional:
# model: sonnet             # Can be specified but not required
---
```

**File Locations:**
- Simple agents: `.claude/agents/{name}.md`
- Workers: `.claude/agents/{domain}/workers/{name}.md`
- Orchestrators: `.claude/agents/{domain}/orchestrators/{name}.md`
- Skills: `.claude/skills/{skill-name}/SKILL.md`
- Commands: `.claude/commands/{command-name}.md`

**Worker Requirements:**
- Phase 1: Read Plan File (`.tmp/current/plans/.{workflow}-plan.json`)
- Phase 2: Execute Work
- Phase 3: Validate Work
- Phase 4: Generate Report (use `generate-report-header` skill)
- Phase 5: Return Control
- Must reference: `run-quality-gate`, `generate-report-header`, `rollback-changes`

**Orchestrator Requirements:**
- Return Control Pattern (signal readiness → exit → resume by main session)
- Quality gates with blocking logic
- TodoWrite progress tracking
- Plan file validation (`validate-plan-file` skill)
- NO Task tool to invoke workers (FORBIDDEN)
- Must reference: `validate-plan-file`, `run-quality-gate`, `rollback-changes`

**MCP Integration Guidance:**
- Orchestrators SHOULD include MCP guidance in plan files
- Workers MUST use MCP for implementation when relevant
- Context7 (`mcp__context7__*`) for library patterns
- Supabase (`mcp__supabase__*`) for DB operations (requires `.mcp.full.json`)
- n8n (`mcp__n8n-mcp__*`) for workflow management

## 6. Alignment Gaps and Recommendations

### Critical Gaps Identified

1. **RBAC/ABAC Implementation Agent**
   - **Gap:** While `database-architect` defines RLS and `api-builder` handles auth middleware, there's no dedicated agent for implementing role-based/attribute-based access control logic across the stack.
   - **Impact:** Complex RBAC rules (e.g., "HR director can view all courses in their org, but not other orgs") require a dedicated implementation pattern.
   - **Recommendation:** Propose `rbac-implementer` worker (see section 7).

2. **SSO/SAML Integration Specialist**
   - **Gap:** `api-builder` handles general auth, but SSO/SAML (required for Enterprise tier per roadmap) needs specialized configuration (IdP metadata, SP configuration, certificate management).
   - **Impact:** Enterprise customers blocked without SSO.
   - **Recommendation:** Propose `sso-integration-specialist` simple agent (see section 7).

3. **Observability Orchestrator**
   - **Gap:** `infrastructure-specialist` sets up services, `logging-and-audit` defines strategy, but no orchestrator exists for end-to-end observability rollout (metrics, traces, dashboards, SLA alerts).
   - **Impact:** SLA monitoring requirement (roadmap #6) not fully addressed.
   - **Recommendation:** Propose `observability-orchestrator` (see section 7).

4. **LLM Evaluation Orchestrator**
   - **Gap:** `ai-automation` designs prompts and evals, but no orchestrator exists for running regression suites, A/B tests, and canary rollouts for LLM features.
   - **Impact:** AI quality assurance is manual; no systematic eval workflow.
   - **Recommendation:** Propose `llm-eval-orchestrator` (see section 7).

5. **Competency Matrix Implementation**
   - **Gap:** `hr-people-ops` creates competency matrices, but no agent implements them as code (e.g., mapping roles to required skills in DB, generating skill gap reports).
   - **Impact:** HR processes remain document-only, not integrated with LMS.
   - **Recommendation:** Consider extending `database-architect` or creating HR-specific implementation agent (lower priority).

### Non-Blocking Observations

1. **Spec Workflow Coverage:**
   - `product-manager` coordinates speckit commands and KFC agents well. No gaps.

2. **Overlap with Existing Agents:**
   - No duplicates detected. New agents fill distinct roles.

3. **MCP Dependency:**
   - `system-architect`, `logging-and-audit`, `ai-automation` all reference Context7 MCP correctly.
   - `ai-automation` uses optional n8n-mcp, which is appropriate.

## 7. Future Agents Proposed

To close the gaps identified above, the following agents are recommended for future development:

### 1. rbac-implementer (Worker)

**Purpose:** Implement RBAC/ABAC policies across database (RLS), API (middleware), and UI (route guards) based on role definitions from `system-architect` and `database-architect`.

**Type:** Worker
**Domain:** `security`
**File Location:** `.claude/agents/security/workers/rbac-implementer.md`

**Responsibilities:**
- Read plan with role definitions and permission rules
- Generate RLS policies for Supabase tables
- Generate tRPC middleware for route-level authorization
- Generate UI route guards (Next.js middleware)
- Log all changes for rollback
- Validate via `run-quality-gate` (type-check, tests)
- Generate structured report

**Skills:** `run-quality-gate`, `generate-report-header`, `rollback-changes`, `parse-package-json`
**MCP:** Supabase (for RLS policy deployment), Context7 (for auth patterns)
**Orchestrator:** Security-related orchestrator or invoked by `system-architect`

### 2. sso-integration-specialist (Simple Agent)

**Purpose:** Configure SSO/SAML integration for Enterprise customers (IdP metadata, SP configuration, certificate management, testing).

**Type:** Simple Agent
**File Location:** `.claude/agents/sso-integration-specialist.md`

**Responsibilities:**
- Gather IdP metadata and SP requirements from user
- Generate Supabase Auth SAML configuration
- Provide certificate management checklist
- Generate test plan for SSO flow (happy path, error cases)
- Output integration guide for ops team

**Skills:** `render-template`, `format-markdown-table`, `generate-report-header`
**MCP:** Context7 (for SAML patterns), Supabase (for auth config)
**Collaboration:** `api-builder`, `infrastructure-specialist`

### 3. observability-orchestrator (Orchestrator)

**Purpose:** Orchestrate end-to-end observability rollout: logging strategy → metrics → traces → dashboards → SLA alerts.

**Type:** Orchestrator
**Domain:** `infrastructure`
**File Location:** `.claude/agents/infrastructure/orchestrators/observability-orchestrator.md`

**Phases:**
1. **Pre-Flight:** Setup `.tmp/current/` structure

3. **Phase 2:** Metrics instrumentation (worker TBD)
4. **Phase 3:** Trace propagation (worker TBD)

6. **Phase 5:** SLA alert configuration (worker TBD)
7. **Quality Gate:** Validate all instrumentation compiles and basic smoke tests pass
8. **Summary:** Generate observability readiness report

**Skills:** `validate-plan-file`, `run-quality-gate`, `rollback-changes`, `format-todo-list`
**MCP:** Context7 (for OpenTelemetry patterns)
**Workers:** `logging-and-audit`, plus future workers for metrics, traces, dashboards

### 4. llm-eval-orchestrator (Orchestrator)

**Purpose:** Orchestrate LLM evaluation workflow: prompt changes → regression suite → A/B test → canary rollout.

**Type:** Orchestrator
**Domain:** `ai`

2. **Phase 1:** Baseline eval (worker runs golden set)

**Workers:** Future eval workers (baseline-evaluator, regression-checker, canary-monitor)
